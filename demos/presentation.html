Q1

为什么 Token 计费不一样？

同一句话，三家模型的 Token 数不同——有人在偷工减料吗？

🤖 DeepSeek

~5

"今天天气真好" 的 Token 数

切得粗 → Token少 → 单价低但可能语义粗糙

🌟 Kimi

~7

"今天天气真好" 的 Token 数

切得细 → Token多 → 对中文更友好但算力开销大

🧠 OpenAI

~6

"今天天气真好" 的 Token 数

中间路线 → 英文出身，中文适中

🔪 Token 就像是"切菜"——同一颗白菜，有人切大块、有人切细丝

想象你要把一根黄瓜切成段：有人喜欢大段（一刀三段），有人喜欢切片（一刀二十片）。
同一根黄瓜，切出来的块数不同，但黄瓜还是那根黄瓜。

LLM 的 Token 也一样："今天天气真好"这5个字，不同厂商的"切法"不同。
DeepSeek 可能整段切"今天/天气/真好"，Kimi 可能细切"今/天/天/气/真/好"。
字没变，切法变了 → Token 数量变了 → 计费自然不一样。

Token 不是"字"，而是厂商定义的最小语义单元。不同词表 = 不同切分 = 不同数量 = 不同计费。

🔑 三个根本原因

训练数据不同

如果 90% 训练数据是英文，中文就会切得更碎 → Token更多

词表大小策略不同

大词表=更少Token=更快，但更占显存（下一页详解 →）

商业策略

按Token收费 → Token定义直接影响利润和用户感知

💡 一句话记住：就像不同刀工做同一道菜，食材没变，块数不一样。Token 是模型"吃"进去的食物，切大块（少Token）省咀嚼次数，切细碎（多Token）更易消化。

根因二

词表大小 × 向量维度 → 显存 & 速度的博弈

为什么"大词表 = 更少 Token = 更快，但更占显存"？

📚 词表就是一本"字典"——词条数和释义宽度是两个独立维度

想象一本字典：
· 词表大小（vocab_size） = 字典有多少个词条，比如 5000 条还是 100000 条
· 向量维度（hidden_dim） = 每个词条的释义栏位有多宽，写 3 行还是写 30 行

两者完全独立：
一本 50000 词条的字典，每页可以写 3 行（dim=768），也可以写 30 行（dim=4096）。
一本 1000 词条的袖珍字典，同样可以写 3 行或 30 行。

页数不决定每页写几行，反之亦然——但页数 × 每页宽度，就是这本字典的"纸张总面积"。
在 LLM 里，这个总面积 = Embedding 矩阵占用的显存。

🔢

vocab_size（词表大小）

决定者：分词器

BPE 训练时合并了多少轮
基础字符 + 合并次数 = 最终词表
例：Llama 3 ≈ 256 + 128000次

影响：Token 数量和推理速度
大词表 → Token 少 → 序列短 → 快
📐

hidden_dim（向量维度）

决定者：架构设计

模型设计师根据目标参数反推
要与头数、层数配合
例：4096 维 → 8B 参数档次

影响：每层参数量 = O(d²)
d翻倍 → 参数 ×4 → 显存爆炸
✖️

两者的乘积

= Embedding 显存

vocab_size × hidden_dim × 2 bytes
这就是 Embedding + LM Head
两个矩阵 = 总参数 ~15%

权衡：词表大 & 维度大 不可兼得
需要找到工程的甜蜜点
Embedding 显存 = vocab_size × hidden_dim × 2 (FP16)  ×  2 (输入+输出矩阵)
≈ vocab_size × hidden_dim × 4 字节  |  这是需要"常驻显存"的最小开销，还不算 Transformer 层

Q1 序列是什么？为什么 Token 少 = 序列短？

序列 = Token ID 排成的一行数字。模型读到的不是汉字，而是编号列表：

字符级 → [89, 234, 56, 12, 190] n = 5
BPE级 → [896, 23401] n = 2

序列长度 n = Token 数量，所以"Token 少 → 序列短"是同义反复——
序列的唯一度量就是 Token 个数。

Q3 KV Cache 是什么？为什么也变少？

生成时（一字一字吐），每吐一个新 Token，都要回头"看"之前所有 Token：

❌ 没有 KV Cache
第1步: 算 t₁ 的 K/V
第2步: 重新算 t₁ + 算 t₂ 的 K/V
第3步: 重新算 t₁,t₂ + 算 t₃ 的 K/V
大量重复计算！
✅ 有 KV Cache
第1步: 算 t₁ → 存 K₁/V₁
第2步: 读 K₁/V₁ + 算 t₂ → 存 K₂/V₂
第3步: 读 K₁₂/V₁₂ + 算 t₃ → 存 K₃/V₃
算过的不再重算
Cache 大小 = n × hidden_dim × 2(K+V) × 层数 × 2 bytes
n ↓ → Cache 正比例缩小 → 同样显存撑更大 batch → 吞吐量提升

Q4 推理为什么变快？三重加速叠加

🚀 加速一：Attention 计算量平方级缩减

n 从 1000 → 500，计算从 1,000,000 → 250,000，减少 75%

📦 加速二：KV Cache 小 → Batch Size 更大

相同显存下原本只能并行 4 条请求，现在能处理 8 条 → 吞吐量翻倍

⚡ 加速三：KV Cache 小 → Memory I/O 减少

推理瓶颈往往是显存带宽而非算力。Cache 越小，每步读回数据越少
读 2GB(1ms) vs 读 4GB(2ms) → 每 Token 省 1ms → 生 1000 Token 省 1 秒

💡 一句话记住：大词表让每个 Token "更重"（覆盖更多语义）→ 序列短 → Attention O(n²) 骤降 → KV Cache 小，显存省 + batch 更大 + Memory I/O 更少 → 三重加速系统性提升推理效率。用 Embedding 层几百 MB 显存换全链路平方级缩减，稳赚不赔。

BPE

BPE 分词器原理

Byte Pair Encoding —— 从字符开始，反复合并最常见的相邻对，直到词表够用

📖 以 "北京北京天安门" 为例的 BPE 合并过程

Step 0 · 按字拆分 (vocab_size=5)
北京北京天安门

先当每个字都是独立Token → 统计相邻对出现次数 → 找最高频的那一对

↓ "北京" 出现 2 次 (最高频) → 合并为新Token "北京" ↓
Step 1 · "北京"合并 (vocab_size=6)
北京北京天安门

vocab 新增 "北京" → 这两个字以后就是一家人了 → 继续统计下一个高频对

↓ "天安" 出现 1 次 → 合并为 "天安" ↓
Step 2 · "天安"合并 (vocab_size=7)
北京北京天安门

... 重复此过程，直到词表达到目标大小（如 50000）

🧱 BPE 就是"搭乐高"——常见结构提前拼好，罕见的现场搭

想象你有一套乐高：基础积木是单个字（北、京、天、安、门、今、天、气）。
你发现 "北京"这两个积木总是一起出现，于是把它们粘成一个新积木。
你又发现 "天安" 也经常连在一起，也粘起来。最后 "今天"、"天气" 也都粘了。

结果：高频词 "北京" 变成一个积木（快），低频词 "捭阖" 还得用 "捭"+"阖"（灵活）。
这套积木的总种类数（vocab_size）是你人为设定的——比如最多 50000 种。

✅ BPE 的精妙之处

高频词保持完整 → 1个Token → 效率高
低频词拆成子词 → 灵活组合 → 不浪费
未登录词自动拆为已知子词 → 永远不OOV
多语言无需语言特有规则 → 字节级通用

💡 一句话记住：BPE 就是统计"哪些字总在一起出现"，然后把它们打包成一个单位。就像你不会把 "巧克力" 拆成 "巧"+"克"+"力" 三个字来读，BPE 也不会。

📦 Qwen

Dense 全规格

0.5B ~ 72B 全覆盖
从手机到数据中心，总有一款适合你

灵活选择
① Token ID → ② Embedding → ③ +RoPE → ④ Attn×N → ⑤ FFN×N → ⑥ Linear → ⑦ Next Token
① 分词 Token IDs → 入口编号

文字被切成一串数字 ID，如 "北京天安门" → [896, 23401]。这是流水线的唯一输入格式，模型只能"读"数字。

② 向量化 Embedding → 查表变向量

每个 ID 查表得到一个 4096 维向量，把离散编号变成有语义距离的坐标。"北京"和"上海"的向量会比"北京"和"苹果"更接近。

③ 位置 +RoPE → 注入"前后"信息

Self-Attention 分不清 "狗咬人" 和 "人咬狗"。RoPE 用旋转矩阵给每个位置打上角度标签，让模型知道谁在前、谁在后。

🎯 Kimi 改装点 —— 训练用 32K 角度，推理时用数学公式外推到 200 万
④ 交流 Attention × N → 互相"开会"

每个 Token 计算 Q·Kᵀ 决定"该听谁的"，再按权重加权 V。N 层叠起来，底层学语法（形容词修饰什么），高层学语义（这句话在说什么）。

⑤ 加工 FFN × N → 独立"思考"

Attention 做完信息交换后，FFN（两层 MLP + 激活函数）对每个 Token 做深度非线性变换，把"开会讨论的结果"沉淀为自己的理解。

🎯 DeepSeek 改装点 —— FFN 改成 8 选 2 MoE，671B 参数每次只激活 37B
⑥ 投影 Linear → 映射到词表

把最后一层的 hidden_dim 维向量投影到 vocab_size 维，每个维度代表一个候选词。分数越高 = 模型越倾向选这个词。

⑦ 决策 Softmax → 概率选词

Linear 的原始分数经过 Softmax 归一为 [0,1] 概率分布，选概率最高的 Token 输出。然后这个 Token 追加到序列末尾，从头再走一遍流水线。

改装 ⑤ FFN 为 MoE 稀疏激活
671B 参数 → 每次只跑 37B
📏 Kimi
改装 ③ RoPE 做位置外推
训练 32K → 推理扩展至 200 万
📦 Qwen
不改流水线环节，改 尺寸参数

Attn

Self-Attention：每个 Token 都在"开会"

Attention 让每个词关注所有其他词，决定"该听谁的"

🏢 Attention 就像一场会议——每个人在听所有人说话，但重点不同

想象一个会议室里坐着四个人：北京、今天、天气、好。
轮到 "好" 发言时，它会扫视全场：
· 听"北京"：注意力 0.1（谢谢，但不太相关）
· 听"今天"：注意力 0.1（稍微听一下）
· 听"天气"：注意力 0.4（哦，"天气好"，这个很相关！）
· 听自己"好"：注意力 0.4（自己的信息也不能丢）

最后 "好" 综合所有人的发言（按注意力权重加权），形成自己的理解。
每个词都做同样的事 → 这就是 Self-Attention。

🔍 Q (Query) - 查询

"我在找什么？"
就像你在图书馆检索系统里输入关键词
每个Token生成一个问题，去匹配别人

🔑 K (Key) - 索引

"我能提供什么？"
就像每本书封底的标签和简介
供别人来快速判断是否相关

💎 V (Value) - 内容

"我实际给你什么信息？"
就像书的正文内容
匹配上了就传递实际的知识

🧮 Attention 计算 = Q和K算匹配度 × V取信息

1

投影：每个Token × 三个矩阵 → 得到自己的 Q, K, V（三个不同身份）

2

打分：Q × K^T → 一个分数矩阵，表示"我对你有多感兴趣"

3

归一化：除以 √d 防止分数过大 → Softmax 转为概率（每行加起来=1）

4

加权：概率 × V → 按"感兴趣程度"加权取信息 → 输出

"Attention 不复杂：每个词都向其他词提问（Q），别人提供标签（K），聊得来就多听几句（V）。除以 √d 只是防止嗓门太大听不见别人说话。"

RoPE

RoPE 旋转位置编码 + MoE 混合专家

位置编码的数学之美 × 稀疏激活的成本魔法

🔄 RoPE：用"角度"表示"位置"

🕐 RoPE 就像钟表指针——用角度差表示相对距离

想象一个钟：3点钟的指针指向 90°，6点钟指向 180°。
你不需要知道"现在是几点"，你只需要知道 两个指针之间的角度差（90° = 3小时）。

RoPE 把每个 token 变成一个向量，然后按它在句子中的位置旋转一个角度：
第0个词 → 不旋转（θ=0°），第1个词 → 转 θ，第2个词 → 转 2θ...
两个词之间的角度差 = (pos2 - pos1) × θ，天然蕴含了相对位置！
传统编码记住"我在第5位"，RoPE 只需要知道"我离他差3个位置"。

角度差 = 位置差 × θ → 相对位置自动蕴含在旋转中

传统绝对编码

训练最长512 → 推理512以上就乱
"我只会数到512"

RoPE 相对编码

可以外推到训练时没见过的长度
"我算角度差，不需要数数"

→ Kimi 利用 RoPE 特性，把 32K 外推到 200 万 token 上下文

🧩 MoE：每次只激活一部分参数

🏥 MoE 就像医院分诊——不同问题挂不同科室

你感冒了去医院，不是所有医生都来看你：

核心思想：把绝对位置信息编码为向量的旋转角度
巧妙之处：两个向量的内积只依赖角度差 → 自动蕴含相对位置
为什么能外推：角度是连续的，训练后可以继续"转"到没见过的大角度

🧩 MoE 本质

核心思想：很多专家，但每次只叫需要的几个
巧妙之处：参数总量可以极大，但计算量只和激活的专家有关
为什么省钱：671B参数只需37B的算力 → 大模型的能力，小模型的成本

Q3

为什么同一问题两次答案不同？

LLM 不是搜索引擎，是"概率预测机" —— 每次都在抽签

🎰 LLM 生成 = 每步一次"加权抽签"，不是查表

想象你有一个骰子，但不是6面均匀的——"2"那一面被削大了（85%概率），
"两"、"二"、"贰" 各占一点（5%, 4%, 2%）。

每次要输出一个答案，你掷这个骰子，而不是直接翻书查答案。
大多数时候会掷出 "2"（因为概率最高），但偶尔也会掷出 "两" 或 "贰"。

这就是为什么同一问题问两遍，答案可能略有不同：两次掷骰子的结果不一定一样。

🎰 逐Token生成 —— 一个不断"掷骰子"的过程

流程 # 每一步都是：算概率 → 按策略选一个 → 拼回去再算下一步
输入 "1+1="
↓ 模型计算下一个 token 的概率分布
{"2": 0.85, "两": 0.05, "二": 0.04, "贰": 0.02, ...}
↓ 根据策略选一个 → 假设选了 "2"（概率最大）
↓ 输入变成 "1+1=2"，继续预测下一个 token
{"。": 0.7, "，": 0.1, "亲": 0.02, ...}
↓ 如果选了 "亲"（低概率冷门事件发生了！）
输出 "等于2，亲" ← 两次结果不同就发生在这步

🌡️ Temperature（温度）

"辣椒"参数——控制答案有多"敢冒险"

值效果类比
0.1几乎总选最高概率不放辣椒——清淡稳定
0.7适中随机微辣——有变化但不过分
1.5低概率也有机会变态辣——什么都有可能

🎯 Top-p（核采样）

"入围线"——只从头部候选里选

值效果类比
0.1只在最可能的10%里选只看前三名
0.550%候选参与看前十名
0.990%候选参与 → 多样看前五十名

📊 Top-k

"名额限制"——最多k个候选人

值效果类比
1010选1 → 很集中只面试10个人
5050选1 → 适中面试50个人
100100选1 → 很多样面试100个人
