Q1

为什么 Token 计费不一样？

同一句话，三家模型的 Token 数不同——有人在骗钱吗？

🤖 DeepSeek

~5

"今天天气真好" 的 Token 数

🌟 Kimi

~7

"今天天气真好" 的 Token 数

🧠 OpenAI

~6

"今天天气真好" 的 Token 数

Token 不是"字"，而是厂商定义的最小语义单元。不同词表 = 不同切分 = 不同数量。

🔑 核心原因

训练数据不同
在英文数据上训练的分词器切中文更碎

词表大小策略不同
大词表 = 更少 Token = 更快，但更占显存

商业策略
Token 计费直接影响成本和利润

BPE

BPE 分词器原理

Byte Pair Encoding —— 从字符开始，反复合并最常见的相邻字符对

📖 以 "北京北京天安门" 为例的 BPE 合并过程

Step 0 · 初始词表 (5 字符)
北京北京天安门

vocab = {"北", "京", "北", "京", "天", "安", "门"} (5个唯一) → 传入 Step 1

↓ "北京" 出现 2 次 → 合并为 "北京" ↓
Step 1 · vocab_size=6
北京北京天安门

vocab 新增 "北京" → 继续查找下一个高频对

↓ "天安" 出现 1 次 → 合并为 "天安" ↓
Step 2 · vocab_size=7
北京北京天安门

... 重复直到词表达到目标大小

✅ 最终效果

高频词"北京"保持为一个 token — 效率高
低频词"天安"拆更碎 — 灵活不浪费

🎯 为什么不同

vocab_size 是人为设定的超参数
合并次数不同 → 同一文本切出的 token 数量不同

🔍 Q (Query) - "我在找什么？"

每个词向所有词"提问"
决定自己对谁感兴趣

🔑 K (Key) - "我能提供什么？"

每个词展示自己的"标签"
供其他词来匹配

💎 V (Value) - "我实际给你什么？"

经过注意力权重加权后
最终输出的实际内容

Q · K^T → 注意力分数 → Softmax → 加权 V → 输出。除以 √d 防止梯度消失。

RoPE

RoPE 旋转位置编码 + MoE 混合专家

Kimi 的长上下文和 DeepSeek 的低成本秘密

Q3

为什么同一问题两次答案不同？

LLM 不是搜索引擎，是"概率预测机"

🎰 LLM 生成 = 每步一次概率抽奖

流程 # 不是：数据库检索 → 返回固定答案 # 而是：逐 token 预测下一个最可能的 token
输入 "1+1="
↓ 模型计算下一个 token 的概率分布
{"2": 0.85, "两": 0.05, "二": 0.04, "贰": 0.02, ...}
↓ 根据策略选一个 → 假设选了 "2"
↓ 输入变成 "1+1=2"，继续预测下一个 token
{"。": 0.7, "，": 0.1, "亲": 0.02, ...}
↓ 如果选了 "亲"
输出 "等于2，亲"

🌡️ Temperature

温度越高，概率分布越"平坦"

值效果
0.1几乎总选最高概率 → 确定性
0.7适度随机 → 平衡
1.5低概率也有机会 → 高度随机

🎯 Top-p

只从累计概率达 p 的候选中选

值效果
0.1只在最可能的 10% 里选
0.550% 候选参与
0.990% 候选参与 → 多样性高

📊 Top-k

只保留概率最高的 k 个候选

值效果
1010 选 1 → 很集中
5050 选 1 → 适中
100100 选 1 → 很多样

LLM 不知道"1+1=2"，它只知道在中文里 "1+1=" 后面出现 "2" 的概率最高。Temperature 就是你的确定性旋钮。
